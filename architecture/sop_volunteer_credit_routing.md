# SOP — Salesforce Volunteer Credit Export Routing

**Owner:** `parse_volunteer_credit.ts`
**Spec source:** CLAUDE.md §4.3, §6.2.1, §7.3 — superseded by 2026-05-20 + 2026-05-22 routing decisions (see Rationale)
**Status:** Locked 2026-05-22 (multi-team filter + multi-dimensional opportunity-name routing)

---

## Purpose

Parse the Salesforce-generated Volunteer Credit Export(s) and route each row to one or more metric dimensions per the rules in `sop_metrics_and_good_standing.md`.

## Rationale — why this SOP supersedes the literal §4.3 spec

CLAUDE.md §4.3 originally described a unified single-row shape with `Full Contact ID | Contact Full Name | Opportunity: Opportunity Name | Amount Credited | Type | Volunteer Job: Volunteer Job Name | Volunteer Job: Campaign`.

The actual file produced by the Salesforce report is **two side-by-side blocks in one sheet**. Until 2026-05-22 we assumed "one file per team mascot." That was wrong: Conor's v2.0 export carries **multiple teams in a single file**, with the team list encoded inside the report's filter-descriptor rows. CLAUDE.md §4.3 will be updated in a follow-up edit to match.

Key reconciliations:
- "One file per team mascot" → **dropped.** One file may cover multiple teams.
- "Route each row by Type + Campaign" → **multi-dimensional opportunity-name routing.** Each opportunity name maps to an *array* of metric dimensions (`routing_metrics`).
- "Full Contact ID required" → still required.

## Input contract

| File | Source | Cadence |
|---|---|---|
| `{period-context}-{version}-{timestamp}.xlsx` (one file, multiple teams) | Conor downloads from Salesforce → drops into shared directory | Manual / scheduled (frequency TBD) |

**Filename pattern (locked 2026-05-22):**
- Examples: `2025-26 Active YJ - 2.0-2026-05-22-11-40-54.xlsx`, `2026-27 Active YJ - 1.0-{timestamp}.xlsx`.
- No team mascot in the filename. Period context (`2025-26 Active YJ`) + version (`- 2.0`) + trailing `-YYYY-MM-DD-HH-MM-SS.xlsx`.
- Version (`- 2.0`) is a log/diagnostics tag — not a control signal. Engine does not branch on version.
- Discovery is by sheet content (block labels on row 19) plus filename ending in `-YYYY-MM-DD-HH-MM-SS.xlsx`. See `sop_orchestrator.md` § Source-file discovery.

The team scope of the file is **read from the filter rows inside the sheet**, not parsed from the filename.

## File structure (observed 2026-05-22 in `2025-26 Active YJ - 2.0-2026-05-22-11-40-54.xlsx`)

| Row | Content |
|---|---|
| 2 | Report title (e.g. `2025-26 Active YJ - 2.0`). |
| 3 | "As of {timestamp} Mountain Standard Time/MST • Generated by {operator}". |
| 6 | `Filtered By` label. |
| 7 | `Yellow Jacket Credit- Opportunities` (left-block filter section header). |
| 8 | `Show: All accounts`. |
| **9** | **Left-block date filter** — `Date Field: Opportunity: Close Date equals Custom (MM/DD/YYYY to MM/DD/YYYY)`. Used for historical-vs-current detection. |
| 10 | `Account Name contains Yellow Jacket`. |
| **11** | **Left-block team filter** — `Yellow Jacket Team equals {Team1},{Team2},…`. Comma-joined team mascots. |
| 12 | `Volunteer Points` (right-block filter section header). |
| 13 | `Show: All accounts`. |
| **14** | **Right-block date filter** — `Date Field: Planned Start Date & Time equals Custom (MM/DD/YYYY to MM/DD/YYYY)`. |
| 15 | `Account Name contains Yellow Jacket`. |
| **16** | **Right-block team filter** — `Yellow Jacket Team equals {Team1},{Team2},…`. Should match row 11. |
| 19 | Block label row: `Yellow Jacket Credit- Opportunities` (col B) | `Volunteer Points` (col H). |
| **20** | **Header row.** |
| 21 → N | Data rows. |
| N+1 → end | One `Total` row per block, then `Confidential` + copyright footer. **Ignore.** |

> The header row index varies across export versions. The parser locates it by scanning rows 1–60 for both block-anchor labels (`Opportunity: Opportunity Name` AND `Volunteer Job: Volunteer Job Name`) on the same row. It then walks backward from the header to capture filter rows.

### Header row 20 (production target)

| Block | Columns | Required |
|---|---|---|
| **Left — Opportunities (dollars)** | `Contact Full Name`, `Full Contact ID`, `Opportunity: Opportunity Name`, `Amount Credited`, `Type` | Yes |
| **Right — Volunteer Points (points)** | `Contact Full Name`, `Full Contact ID`, `Volunteer Job: Volunteer Job Name`, `Volunteer Points`, `Volunteer Job: Campaign` | Yes |

Parser identifies blocks by matching the header label strings, not by fixed column letters.

## File-level metadata extracted by the parser

```ts
type CreditFileMetadata = {
  source_file: string;
  source_file_fingerprint: string; // sha256(file content), first 16 chars
  source_file_timestamp: string;    // trailing YYYY-MM-DD-HH-MM-SS from filename
  version_label: string | null;     // e.g. "2.0" from "- 2.0-"; diagnostics only
  teams_in_filter: string[];        // union of row-11 and row-16 team lists
  opportunities_date_range: { start: string; end: string } | null; // from row 9
  points_date_range: { start: string; end: string } | null;        // from row 14
};
```

`teams_in_filter` is computed as the **union** of row-11 and row-16 team lists. If the two rows disagree (Codex adversarial concern), the union is taken and a `credit_filter_team_disagreement` ingest warning is emitted naming both lists — credit still flows from both blocks. Disagreement is a data-quality flag, not a fatal.

`opportunities_date_range.end` (parsed from row 9) is the **authoritative** date used for historical-vs-current detection (the points block's date range is for diagnostics). See `sop_historical_baseline_ingest.md`.

## Multi-dimensional routing (locked 2026-05-22)

Each opportunity name maps to a `routing_metrics` array — the set of metric dimensions that receive the full `amount_dollars` (and proportional `totalPoints` via the points multiplier). Mirrors the additive behavior of `sop_item_categorization.md` for Unify items.

### Routing precedence (per row)

```
1a. Exact ignore seed — volunteer_credit_allowlist row whose
    opportunity_name_pattern matches and whose `routing` column is 'ignore'.
    SHORT-CIRCUIT: drop the row silently. No pattern fallback, no Type fallback,
    no warning. Operator opted to exclude this opportunity by name. (Codex DIM-3)

1b. Exact route seed — volunteer_credit_allowlist row whose
    opportunity_name_pattern matches and whose `routing_metrics` is non-empty
    (or legacy `routing='dollars'` with empty routing_metrics, treated as
    [total_fundraising, total_points]). If found:
    → apply that row's routing_metrics. (Type is ignored.)

2. Pattern match — volunteer_credit_routing_patterns row whose substring pattern
   appears case-insensitively in the opportunity name. If found:
   → apply the first matching pattern's routing_metrics (sort_order ascending).

3. Type-based fallback:
     - Type ∈ {Sponsorship, In-Kind} → [total_fundraising, total_points] (default route to dollars)
     - Type == Cash                  → ingest_errors of kind 'unallowlisted_opportunity_name', skip row
     - Other Type                    → ingest_errors of kind 'unknown_credit_type', skip row
```

**Rationale (Kirk-confirmed 2026-05-22):**
- All three Types use the same name-driven routing. A Sponsorship named `Race for Wishes` adds to wishesForTeachers just as a Cash row with that name does.
- Cash is held to a higher bar because its catch-all default would silently lump miscellaneous Cash credits into totalFundraising. Sponsorship/In-Kind are inherently dollar-denominated revenue and safe to default.
- "Exact wins over pattern" answers Codex's pattern-precedence question: a seed entry for `Race for Wishes` (which routes to `[total_fundraising, wishes_for_teachers, total_points]`) is NOT overridden by a generic `Wishes` pattern, even if the pattern would yield the same routing. Specific declarations win.

### Pilot seed entries (locked 2026-05-22)

Exact-match seeds in `volunteer_credit_allowlist`:

| `opportunity_name_pattern` | `routing_metrics` |
|---|---|
| `2025 FKE Paddle Raise` | `[total_fundraising, total_points]` |
| `Race for Wishes` | `[total_fundraising, wishes_for_teachers, total_points]` |
| `2025 Future Dues` | `[total_fundraising, total_points]` |
| `Fiesta Sports Foundation - Donation` | `[total_fundraising, total_points]` |
| `2025 Par 3 - Silent Auction` | `[total_fundraising, total_points]` |
| `2025 Par 3 - Prize Donation` | `[total_fundraising, total_points]` |

Pattern seeds in `volunteer_credit_routing_patterns`:

| `pattern` (substring, case-insensitive) | `routing_metrics` | `sort_order` |
|---|---|---|
| `Rate Bowl` | `[total_fundraising, rate_bowl, total_points]` | 10 |
| `Wishes` | `[total_fundraising, wishes_for_teachers, total_points]` | 20 |

Notes:
- Future Dues (member-paid dues) count toward `totalFundraising`. Locked by Kirk: members paying dues IS fundraising for this foundation.
- In-Kind credits are routed at **full face value, no cap, no weighting**. A $48,000 In-Kind Prize Donation contributes $48,000 to `totalFundraising` (and 48,000 points).
- Par 3 entries route the same as the Unify `Par 3 Challenge` Item pattern — no Par-3-specific metric exists.
- `2025 FKE Paddle Raise` is the Football Kickoff Event paddle raise. Equivalent of the Unify `Football Kickoff` Item pattern.

### Backward compatibility — legacy `routing` enum

The old `routing` enum column (`dollars` | `ignore`) is preserved on `volunteer_credit_allowlist` for backward compatibility but is **deprecated** in favor of `routing_metrics`. Rows whose `routing_metrics` is non-empty take precedence; rows where `routing_metrics` is `[]` and `routing = 'dollars'` are interpreted as `[total_fundraising, total_points]` for transition. The engine prefers `routing_metrics` when both exist.

## Right block — Volunteer Points (unchanged)

Every right-block row routes to `totalPoints` only. Campaign validation:

| `Volunteer Job: Campaign` | Behavior |
|---|---|
| Contains `Committee Participation Points` (case-insensitive) | Route to `totalPoints`. |
| Anything else | `ingest_errors` of kind `unexpected_points_campaign`, row skipped. |

Right-block points never contribute to `totalFundraising`.

## Normalized output records

```ts
type CreditDollarsRecord = {
  source_file: string;
  source_block: 'opportunities';
  source_row_number: number;
  source_row_hash: string;
  source_file_fingerprint: string;
  source_file_timestamp: string;
  file_metadata: CreditFileMetadata;    // includes teams_in_filter
  full_contact_id: string;
  contact_full_name_raw: string;        // for diagnostics only
  opportunity_name: string;
  amount_dollars: number;
  type: 'Cash' | 'Sponsorship' | 'In-Kind' | string;
  routing_metrics: MetricName[];        // resolved by routing precedence above
};

type CreditPointsRecord = {
  source_file: string;
  source_block: 'volunteer_points';
  source_row_number: number;
  source_row_hash: string;
  source_file_fingerprint: string;
  source_file_timestamp: string;
  file_metadata: CreditFileMetadata;
  full_contact_id: string;
  contact_full_name_raw: string;
  volunteer_job_name: string;
  amount_points: number;
  campaign: string;
};

type MetricName = 'total_fundraising' | 'rate_bowl' | 'wishes_for_teachers' | 'total_points';
```

## Multi-file ingest

Each credit-export file is parsed independently. Records from all files merge into a single stream keyed by `full_contact_id`. Under single-file delivery (the 2026-05-22 production mode) this is a single batch.

### Dedup + supersede

**Dedup key** (single-file mode: still applies):
```
(full_contact_id, source_block, source_row_number, source_row_hash, source_file_fingerprint+timestamp)
```

`source_row_number` is part of the key so two legitimate identical sheet rows for the same volunteer/opportunity/amount aren't collapsed (Codex DIM-1).

When multiple credit-export files appear (e.g., Conor uploads a corrected export), the latest `source_file_timestamp` wins **globally across the entire current-season credit-record stream** — no per-team partitioning under the multi-team filter model. All older batches are silently filtered out at aggregation time per `sop_orchestrator.md` § Superseding batches.

### Same-timestamp ties

If two distinct credit-export files share the same `source_file_timestamp` with different `source_file_fingerprint`, both batches are dropped and a single `ambiguous_credit_batch_timestamp` warning is emitted. Operator resolves by deleting one or amending its filename timestamp.

> **Single-file caveat (2026-05-22):** Under Conor's current single-file delivery cadence, same-timestamp ties are unlikely (a second file would require Conor to manually re-export and rename). The tie-detection logic remains active as a defensive guard.

## File-team vs roster-team mismatch (per-row check)

For every credit record, after resolving `full_contact_id` → roster row:

```
if roster.team is null:            silent OK (Life/Board/Future-without-team)
elif teams_in_filter is empty:     silent OK (filter row absent or unparseable)
elif roster.team in teams_in_filter (case-insensitive): silent OK
else: emit ingest_errors kind=file_team_mismatch, credit still flows
```

The check moved from per-file (legacy: filename mascot must match roster team) to **per-row** (multi-team file: roster team must be inside the filter set). Credit still flows even on mismatch; the warning is a data-quality flag for the operator.

## Validation + error handling

| Condition | Action |
|---|---|
| File missing the `Full Contact ID` column on either block | **Hard fail.** Engine refuses to run. |
| Header row not found (file structure changed) | **Hard fail.** |
| Filter row 11 / row 16 absent or unparseable | Continue. `teams_in_filter = []`. Emit `credit_filter_unreadable` warning. |
| Filter row 11 and row 16 disagree | Continue with the **union**. Emit `credit_filter_team_disagreement` warning. |
| Date-range row 9 / 14 absent or unparseable | Continue. `opportunities_date_range = null`. Orchestrator falls back to current-season ingest (legacy behavior). Emit `credit_filter_unreadable` warning. |
| Row has empty `Full Contact ID` | `missing_full_contact_id_on_credit_row`, skip. |
| Row's `Full Contact ID` not in roster | `unknown_full_contact_id`, skip. |
| Left-block `Amount Credited` not a number, or ≤ 0 | `non_positive_amount_credited`, skip. |
| Right-block `Volunteer Points` not a non-negative integer | `invalid_volunteer_points`, skip. |
| Left-block Cash row with unallowlisted opportunity name (no exact + no pattern match) | `unallowlisted_opportunity_name`, skip. |
| Left-block row with unknown Type (not Cash/Sponsorship/In-Kind), no name match | `unknown_credit_type`, skip. |
| Right-block row with unexpected Campaign | `unexpected_points_campaign`, skip. |
