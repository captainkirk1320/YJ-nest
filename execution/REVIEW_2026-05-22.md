## Drift found (fixed this session)

### D1 — `file_team_mismatch` not actually checked in credit-export parse

**SOP:** [sop_volunteer_credit_routing.md § File-team vs roster-team mismatch](../architecture/sop_volunteer_credit_routing.md):

> For every credit record, after resolving full_contact_id → roster row, compare the roster's team to the team mascot in the filename.
> Mismatch → `ingest_errors` of kind `file_team_mismatch`. Credit still flows; mismatch is a data-quality flag.

**Engine state:** [parse_volunteer_credit.ts](src/parse_volunteer_credit.ts) resolves `full_contact_id` to a roster row (for existence) but never compares the roster's `team` to the filename-derived `team_mascot_from_filename`. The check is silently absent.

**Corrective edit:** After successfully resolving a credit row, compare `roster.by_full_contact_id.get(fcid)?.team` to `team_mascot` (case-insensitive). If mismatch, emit `file_team_mismatch` with both values in `detail`.

### D2 — `file_team_mismatch` kind misused for superseded-batch dropping

**SOP:** the SOP-defined meaning of `file_team_mismatch` is filename-mascot vs roster-team mismatch on a per-row basis. Superseded batches are a different concept (per [sop_orchestrator.md § Superseding batches](../architecture/sop_orchestrator.md)).

**Engine state:** [compute_metrics.ts:181-189](src/compute_metrics.ts#L181-L189) emits `file_team_mismatch` with `reason: 'superseded_by_newer_export'` when dropping older batches. This conflates two distinct error semantics and would confuse operators triaging warnings.

**Corrective edit:** Drop the warning emit entirely for superseded batches. Per the orchestrator SOP, superseded batches are an idempotency mechanic, not a data-quality flag — the operator does not need to triage a newer export superseding an older one. Compute_metrics already filters them silently; the warning was unnecessary.

This means [codex_regressions.test.ts:166-170](tests/codex_regressions.test.ts#L166-L170) needs a corresponding update — it currently asserts `file_team_mismatch` is emitted on supersede. The test's *intent* (verify supersede filters correctly) is preserved; we just stop asserting the warning kind.

## Codex adversarial review (2026-05-22)

Routed through `codex:rescue` after D1/D2 fixes. Codex's brief covered the
7 failure-mode categories enumerated in `memory/project_current_phase_handoff.md`.
Four IMPORTANT findings surfaced (no CRITICAL). All fixed this session with
regression coverage in [tests/codex_regressions.test.ts](tests/codex_regressions.test.ts).

### F1 — Tokenizer drops Unify rows when display name contains parens

**Scenario:** `Yellow Jacket Rep = "(2041777) John (Jr.) Smith"`. The `[^,(]+?` class
in the prior regex bailed at the inner `(`, causing zero matches → entire row
dropped with `unparseable_yellow_jacket_rep`.

**Fix:** [parse_unify_csv.ts:28-33,118-148](src/parse_unify_csv.ts) — split on
`,(?=\()` first, then parse each segment with `^\((\d+)\)\s*(.+?)\s*$`. Display
names can now contain anything except a comma-then-paren boundary.

### F2 — Malformed non-numeric tokens silently dropped

**Scenario:** `"(abc) Bad,(123) Good"` previously matched only `(123)` and
proceeded with partial credit to rep 123.

**Fix:** Same tokenizer rewrite — every segment must match the strict pattern
or the WHOLE row fails with `unparseable_yellow_jacket_rep`. Partial credit on
a malformed row is no longer possible.

### F3 — Synthetic volunteer-id collisions not preflighted

**Scenario:** Roster row A has `full_contact_id = "rep_9104"`, Row B has
`sales_rep_id = 9104` with no FCID. Both derive `volunteers.id = "rep_9104"`,
causing silent merge on Supabase upsert.

**Fix:** [parse_master_roster.ts:251-280](src/parse_master_roster.ts) — after
all rows are processed, build a `seenSyntheticIds` map and hard-fail
(`FatalRosterError`) on collision. Also reserves `org_uncredited` against
roster-row capture.

### F4 — Same-timestamp credit-batch ties silently double-counted

**Scenario:** Two distinct credit-export files for one team share
`source_file_timestamp` but have different fingerprints. The
latest-timestamp filter kept both → double-counted dollars and points.

**Fix:** [compute_metrics.ts:163-219](src/compute_metrics.ts) — detect ties by
counting distinct fingerprints per `(team, latest_timestamp)`; on tie, drop
BOTH batches for that team and emit a new
`ambiguous_credit_batch_timestamp` error (added to types.ts +
ingest_errors.ts severity map).
